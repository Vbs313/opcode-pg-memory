二、发现的新边界缺陷
缺陷 1：规则去重使用首行精确匹配，可能误判
位置：apply-reflection.ts 第 120-124 行

问题：当前去重逻辑使用 normalized.includes(whenLine) 检查首行是否已存在于文件中。这存在两个风险：

误去重：两个不同的规则可能有相同的触发条件但不同的动作（如 When 'bash' (output: failed) 下分别处理 npm test 和 pytest 的失败）。当前逻辑会把第二条规则当作重复而跳过。

误漏：相同的规则因格式微调（如多一个空格）不会被识别为重复。

修复建议（非紧急，可入 v3.13.1）：改用规则的全量文本（ruleBullet 整体）做去重 key，而非仅首行。或对 action 部分也做包含检查。

当前影响：低——在当前使用规模下，同一触发条件产生不同动作的概率极低。但需标记为已知限制。

缺陷 2：规则结构化（Active Rules / Archived Rules）未实现
文档要求：P2 包含规则的结构化管理，支持 ## Active Rules 和 ## Archived Rules 区块，rules-injector 仅注入活跃部分。

当前状态：apply-reflection.ts 追加规则到一个扁平的 Automated Rules (from opcode-pg-memory) 区块，无归档分段。rules-injector（由 oh-my-openagent 提供）会读取整个文件，无法区分活跃和归档规则。

影响：低——当前规则总量尚小，未达到膨胀临界点。但这是 P2 的完整定义，当前仅完成了 P2a（去重），P2b（结构化）和 P2c（老化扫描）均未实现。

建议：入 v3.14.0。

缺陷 3：骨架注入降级策略缺失
文档指出的风险：如果项目以函数库为主（无独立文件实体），骨架可能为空。

当前状态：getProjectSkeleton 仅查询 type = 'file'，当无文件实体时返回空字符串，Agent 丧失方向感。

修复建议（可选）：在 file 实体不足 3 个时，降级查询 module 或 function 类型实体作为骨架补充。

缺陷 4：实体名称未规范化
文档指出的风险：UserService 和 userService 会成为两个不同的实体，弱化检索精度。

当前状态：entity-store.ts 唯一约束使用原始 name，未做大小写归一。

建议：在插入前对 name 做小写规范化，保留原始名称作为 display_name 列（需 schema 变更，可入 v4.0）。
