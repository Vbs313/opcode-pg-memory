一、Hermes 的可借鉴设计：三个关键差异
你提出的问题切中了核心。先来看 Hermes 的 skill_manage 工具实际做了什么：

python
# Hermes 的 skill_manage 工具
skill_manage(action='create', name='deploy-k8s', content=...)
  → 验证 name（仅小写字母+数字+连字符, 1-64字符）
  → 创建 skills/<name>/SKILL.md
  → 原子写入（tmp file → rename）
  → 安全扫描
  → 后续会话自动可用
它与我之前建议的「写入 skills/ 目录」在形式上是同一个方向，但 Hermes 有三个更深层的设计值得借鉴：

1. 触发条件：不是「每个 action_plan 都写 skill」，而是「价值过滤」
从 Hermes 的 prompt 源码可以看到：

text
SKILLS_GUIDANCE = (
  "After completing a complex task (5+ tool calls), "
  "fixing a tricky error, or discovering a non-trivial workflow, "
  "save the approach as a skill with skill_manage..."
)
关键触发条件：

5+ 次工具调用的复杂任务（而非单次工具失败）

修复了棘手错误（而非任何 error_pattern）

非平凡工作流（而非所有 workflow）

对照你当前的 apply_reflection：它会对所有 confidence ≥ 0.8 的 action_plan 写入 rules.md。其中大量是宽泛的「bash 输出包含 error → 检查并重试」这类低价值规则，不适合转化为独立 Skill。

2. 自我改进：Skill 不是死的，而是会被「修补」
Hermes 的核心差异不在于「自动创建 Skill」，而在于「使用过程中自动修补」：

"When using a skill and finding it outdated, incomplete, or wrong, patch it immediately with skill_manage(action='patch') — don't wait to be asked. Skills that aren't maintained become liabilities."

这意味着 Skill 文件是一个活的文档，Agent 在使用中发现不适用的步骤会自动更新。这与我们当前 rules.md 的纯追加模式形成根本差异——我们的规则只会增长，从不自我修正。

3. 渐进式披露：Skill 按需加载，不挤占上下文
Hermes 和 Anthropic 的 Agent Skills 标准都遵循 Progressive Disclosure 原则：

第一层：仅注入 Skill 的 name + description（元数据）

第二层：当 Agent 判断需要时，通过 skill_use 或 load_skill 加载完整内容

第三层：Skill 内的 references/ 子文档进一步按需读取

这与 rules.md 的 system.transform 被动注入完全不同——Skill 是被 Agent 主动检索并加载的，只在需要时消耗 token。

二、我们已有的优势：基础设施已成
你已具备 Hermes 需要从头构建的所有底层能力：

能力	Hermes 实现	opcode-pg-memory 实现
经验捕获	memory 工具	observations 表（10K+ 记录）
模式提炼	SKILLS_GUIDANCE 提示词引导	hindsight_reflect 启发式引擎（6 种 pattern）
结构化输出	skill_manage(action='create')	action_plan（JSONB, trigger+action）
自动触发	任务完成后 Agent 自主决策	session.completed 钩子
存储格式	skills/<name>/SKILL.md	rules.md（平面追加）← 待升级
消费方式	Agent 主动 load_skill	system.transform 被动注入
自我改进	skill_manage(action='patch')	❌ 无（P1 的有效性反馈只是在下次反思时降低置信度，不修改已写入的规则）
核心差距仅在于最后三行，而非底层基础设施。

三、方案设计：apply_reflection 双通道输出
基于上述分析，以下是完整方案。

3.1 核心思路：不再一刀切，而是按质量分流
当前：所有 confidence ≥ 0.8 的 action_plan → rules.md
改造后：

text
hindsight_reflect → action_plan
  ├── 低价值规则（单工具、宽泛）→ rules.md（保持现状，平面追加）
  └── 高价值技能（多步骤、跨工具、≥3 步）→ skills/<name>/SKILL.md（新建）
3.2 分流条件
满足以下所有条件时，创建独立 Skill 而非追加到 rules.md：

多步骤：action_plan.steps ≥ 3（从 workflow 类型 pattern 中提取）

跨工具：步骤涉及 ≥ 2 种不同工具

高置信度：confidence ≥ 0.85（比 rules.md 的 0.8 门槛更高）

非已有 Skill：同名 skill 不存在（通过 name 归一化去重）

不满足条件 → 保持当前行为，写入 rules.md。

3.3 Skill 创建逻辑
新增 MCP 工具 create_skill 或扩展 apply_reflection：

text
apply_reflection(reflection_id)
  → 从 reflections 表读取 action_plan
  → 判断分流条件
  → 若满足 → createSkillFromActionPlan(action_plan)
    → 生成 name（从 pattern_type + 触发工具推导）
    → 生成 SKILL.md（YAML frontmatter + 步骤 body）
    → 写入 ~/.config/opencode/skills/<name>/SKILL.md
    → 标记 reflections.applied_at 和 reflections.applied_as = 'skill'
  → 若不满足 → 当前行为（写入 rules.md）
3.4 SKILL.md 生成模板
基于你 action_plan 的数据结构，自动生成的 SKILL.md 示例：

markdown
---
name: fix-failed-pytest
description: When a pytest run fails, run failed tests in isolation with verbose output, check fixture setup, and verify test database state.
license: MIT
metadata:
  generator: opcode-pg-memory
  reflection_id: 5b929479
  created_at: 2026-05-13T10:30:00Z
---

## Trigger

When `bash` output contains `FAILED` from a `pytest` command.

## Steps

1. Run `pytest --lf -x -v` to rerun only the last failed tests with verbose output.
2. If the error is `AssertionError`, inspect the test fixture setup in `conftest.py`.
3. If the error is `OperationalError`, verify the test database is running (`pg_isready`).
4. After fixing, run `pytest --lf -x` again to confirm all previously failed tests pass.

## Edge Cases

- If `pytest --lf` returns 0 failures but original test still appears broken, run full suite with `pytest -x`.
- If `conftest.py` is not the issue, check for `@pytest.fixture` definitions in the test file itself.

## Known Limitations

- This skill assumes pytest is installed and configured as the project's test runner.
- Database-related errors assume PostgreSQL is the backend.
3.5 后续自我改进（P4 的升级形态）
当前 P1 的有效性反馈只是调整置信度，不改已写入的规则。引入 Skill 后，可以升级为真正的「修补」：

Agent 使用 Skill 后发现步骤不适用 → 触发 improve_skill（或 patch_reflection）

该工具读取现有 SKILL.md，追加 Edge Cases 或修改 Steps

更新 metadata.version，保留修改历史

这比 Hermes 的 skill_manage(action='patch') 更可控——因为我们有结构化的 action_plan JSONB 做中间层，而非直接操作 Markdown。

四、与 oh-my-openagent 的集成
4.1 被动注入 → 主动发现
当前 rules.md 通过 system.transform 被动注入。Skill 需要 Agent 主动加载。

oh-my-openagent 已支持：

task({ load_skills: ["fix-failed-pytest"] }) → Agent 在任务启动时加载 Skill

Skill 的 name + description 自动出现在 Agent 的可用工具列表中

这意味着你需要做的最少集成工作只是：将 SKILL.md 写入 oh-my-openagent 扫描的目录。

oh-my-openagent 的 skill 扫描路径：

text
项目级: .opencode/skill/<skill-name>/SKILL.md
全局级: ~/.config/opencode/skill/<skill-name>/SKILL.md
4.2 路径选择
建议：写入项目级 .opencode/skills/auto/<name>/SKILL.md，而非全局。

理由：不同的项目有不同的工作流模式。全局 Skill 适合通用技能（如 git 提交规范），而自动反思产出的 pattern 通常与特定项目绑定。

六、核心总结
你的判断完全正确：将 action_plan 格式化为 Skill MD 文件写入 skills/ 目录，替代当前只写 rules.md 的路径，是弥合与 Hermes 差距的最后一块拼图。

但需要加上分流逻辑——并非所有 reflection 都值得变成独立 Skill。低价值、单工具的规则继续走 rules.md，只有跨工具、多步骤、高置信度的模式才升级为 Skill。

核心设计原则：让 rules.md 做「快速反应规则」，让 skills/ 做「深度工作流资产」——两者互补而非替代。 同时保留将高分规则也留在 rules.md 中的能力（通过双写），确保 Agent 无论通过哪种方式都能接触到关键经验。
