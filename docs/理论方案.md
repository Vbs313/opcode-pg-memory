一、架构总览：六层体系的工程价值
opcode-pg-memory 是目前 OpenCode 生态中唯一具备完整认知闭环（感知→记忆→反思→行动→验证）的插件。其六层架构并非简单的目录划分，而是对应 Agent 认知的六个阶段：

text
Layer 6: Entry      → 神经系统（感知外部事件，分发到各处理单元）
Layer 5: Injection  → 海马体（将记忆编码、压缩、注入工作记忆）
Layer 4: MCP        → 前额叶（执行控制：检索、反思、决策、应用规则）
Layer 3: Hooks      → 感官皮层（截获工具调用、消息、会话事件）
Layer 2: Services   → 皮层下结构（实体提取、因果链、压缩、缓冲）
Layer 1: Types+DB   → 基因/骨架（数据模式、持久化、配置）
这不是比喻——插件的实际数据流严格遵循认知神经科学的层级处理模式：Layer 3 截获原始信号 → Layer 2 提取结构化特征 → Layer 4 进行高维推理 → Layer 5 编码注入 → Layer 6 统合调度。

二、各模块评分与评估
Layer 1: Types + Config（类型配置层）
文件	行数	职责	评分
types.ts	~200	全局类型定义	8.5/10
config.ts	~150	环境配置 + 12 条压缩规则	9/10
db/init-db.ts	~300	Schema 迁移 + pgvector 降级	9/10
架构优势：

config.ts 将 12 条压缩规则外部化为可配置数组（CompressionRule[]），这是“数据驱动设计”的典型案例——修改规则不需要改动压缩引擎。

init-db.ts 实现 fail-open 降级：若 pgvector 扩展不可用，跳过向量列和 HNSW 索引创建，插件仍可正常工作（仅语义搜索不可用）。这与 Claude Code MCP 规范中“tools should handle errors gracefully by returning error responses rather than raising exceptions”的原则一致。

改进建议：

types.ts 中 metadata 字段保留 2 处 Record<string, any>，可进一步约束为 Record<string, unknown> 加类型守卫。

架构优势：

entity-extractor.ts 是最具原创性的模块：零 LLM 纯正则从工具输出提取 file/function/class/module 四类实体，fire-and-forget 非阻塞。这直接否决了“用 LLM 提取实体”的常规方案，节省了大量 token 开销。

entity-store.ts 的 INSERT ON CONFLICT 策略：双部分唯一索引 (name, type, session_map_id) + (name, type) WHERE NULL，完美解决了并发写入竞态问题。

memory-buffer.ts 的事务批量写入：BEGIN → 批量 INSERT → COMMIT 或 ROLLBACK，保证原子性。PG 宕机时不崩溃，本地内存队列 + 指数退避重试。

改进建议：

async-embedder.ts 可引入优先级队列——高重要性 observation 优先向量化，低重要性可延迟或跳过。

privacy.ts 目前实现较简单（仅基本敏感词过滤）。对于需要处理敏感代码的企业场景，可扩展为基于模式匹配的路径/密钥检测。

架构优势：

recall-memory.ts 从 1,532 → 80 行：是本次重构的最大成就。拆分为 6 个检索模块（semantic-search.ts、keyword-search.ts、bm25-search.ts、graph-traversal.ts、fusion.ts、utils.ts），协调器仅保留并行分发、结果融合、排序、过滤逻辑。

hindsight-reflect.ts 的纯启发式引擎：不依赖任何 LLM API，100% 基于正则和统计（tool_status: failed → error_pattern，工具频率 → workflow，编辑-构建-测试序列 → tool_preference）。零 token 消耗、零 API 延迟。

apply-reflection.ts 的五函数管道：checkDuplicate → checkCooldown → formatRule → writeRuleAtomic → markApplied，每步独立可测试。writeRuleAtomic 使用临时文件 + rename 模式，防止并发写入破坏 rules.md。

MCP 工具返回值格式统一：通过 mcp-server.ts 的 respond() 辅助函数，所有 19 个工具返回 { success: true, data } 或 { success: false, error }，符合 Claude Code MCP 规范。

改进建议：

hindsight-reflect.ts 当前仅使用启发式规则。虽然代码中已预留 LLM 路径（performReflectionWithLLM 函数框架），但处于注释状态。如需更高质量的反思（如语义级别错误归类），可通过环境变量 OPCODE_PG_MEMORY_LLM_ENABLED=true 启用。

knowledge-corpus.ts 的 7 个工具中，build_corpus 和 prime_corpus 功能相似，可考虑合并。

Layer 5: Injection（注入层）
文件	行数	职责	评分
system-transform-injector.ts	460	注入协调器	8.5/10
ranking.ts	714	评分排序引擎	9/10
skeleton-injector.ts	70	项目骨架注入	9/10
session-summary-writer.ts	~100	会话摘要落盘	8/10
架构优势：

两路召回 + 混合评分：路径 A（关键词检索，基于 btree 索引，90 天窗口）和路径 B（语义检索，pgvector ANN，3s 超时降级）。混合评分公式 similarity×0.5 + importance×0.3 + recency×0.2，在召回率与准确性之间取得平衡。

渐进式注入：短时记忆层优先（零 PG 查询），冷启动时自动跳过语义检索，避免空库时的无效查询。

skeleton-injector.ts 的 token 预算保护：逐文件追加时检查 estimateTokens(entry) > maxTokens，文件名永不截断，防止产生无效路径污染 Agent 认知。

全局降级机制：当项目级检索结果 <3 条时，自动扩展到全局检索（keywordRecall + reflections + entities），确保冷启动场景下也有记忆可用。

改进建议：

system-transform-injector.ts 目前 460 行，协调器仍包含部分检索管道函数。可进一步将 retrieveMemoriesForInjection 中的本地函数迁移到 ranking.ts，目标 ≤200 行。

ranking.ts 714 行偏大，可拆分为 scoring.ts（评分算法）+ dedup.ts（去重逻辑）+ formatting.ts（格式化输出）。
