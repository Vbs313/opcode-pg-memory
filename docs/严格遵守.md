N2 统一 MCP 工具返回值格式
目的：符合 Claude Code MCP 规范，让任何调用方都能解析。

当前问题：部分工具直接返回 payload，未包装。

操作：
为所有 src/mcp/*.ts 中的工具处理函数添加统一包装器：

typescript
// src/mcp/response-wrapper.ts (新建)
export function success(data: unknown) {
  return { success: true, data };
}
export function failure(error: string) {
  return { success: false, error };
}
然后修改每个工具的主函数，例如 recall_memory：

typescript
try {
  const results = await recallMemoryLogic(input, pool);
  return success(results);
} catch (err) {
  logger.error('recall_memory failed', err);
  return failure((err as Error).message);
}
验证：编写一个简单的集成测试，调用每个工具并检查返回值是否包含 success 字段。

N3 集成 SDK 配置系统
目的：让用户通过 OpenCode 的统一配置界面（opencode.jsonc 或命令行）来设置插件参数，而非环境变量。

操作：

修改 src/config.ts，从 @opencode-ai/plugin 导入 getPluginConfig 函数（如果 SDK 提供）或通过钩子 chat.params 获取配置。

由于 SDK 可能不直接暴露 getPluginConfig，可以采用约定：插件在 index.ts 中接收 PluginInput，其中包含 config 字段。将配置对象存储在模块级变量中，供其他模块使用。

在 index.ts 的插件函数中：

typescript
export const server: Plugin = async (input) => {
  const pluginConfig = input.config || {};
  // 将配置注入到 config 模块
  setRuntimeConfig({
    databaseUrl: pluginConfig.database_url || process.env.DATABASE_URL,
    embeddingProvider: pluginConfig.embedding_provider || 'ollama',
    autoReflect: pluginConfig.auto_reflect !== false,
  });
  // ... 其余初始化
};
更新 opencode.json 中的 config 字段（已在 N1 定义）。

保留环境变量作为回退，但优先使用插件配置。

验证：在 opencode.jsonc 中覆盖配置，确认插件读取的是新值。

N4 实现插件卸载清理
目的：当插件被禁用或 OpenCode 退出时，正确释放资源（数据库连接、定时器）。

操作：

在 index.ts 中返回的 Hooks 对象增加 unload 钩子（如果 SDK 支持）或利用 event 钩子监听 plugin.unload 事件。

根据 SDK 1.14.40 的 Hooks 类型，没有直接的 unload 钩子，但可以在插件函数中返回一个清理函数（需查阅 SDK 文档）。如果不支持，则监听 process.on('exit') 作为后备。

具体代码：

typescript
let pool: Pool | null = null;

export const server: Plugin = async (input) => {
  pool = new Pool({ connectionString: config.databaseUrl });
  // ... 初始化

  // 注册退出清理
  const cleanup = async () => {
    if (pool) {
      await pool.end();
      pool = null;
    }
    logger.info('Plugin cleaned up');
  };

  // 如果 SDK 支持 onUnload 回调（需确认）
  // 否则使用 process 事件
  process.on('beforeExit', cleanup);
  
  return {
    hooks: { ... },
    // 如果 SDK 支持返回清理函数
    // dispose: cleanup,
  };
};
验证：启动插件后，通过 opencode plugin disable opcode-pg-memory 或关闭 OpenCode，检查 PostgreSQL 连接是否释放（pg_stat_activity 中无残留连接）。

N5 适配 SDK 日志通道
目的：让插件日志出现在 OpenCode 的诊断日志中，便于调试。

操作：

修改 src/services/logger.ts，使用 @opencode-ai/plugin 提供的 createLogger 函数（如果存在）或通过 console.error/console.warn 输出，因为这些会被 OpenCode 捕获。

保留现有的结构化日志，但输出重定向到 stderr（MCP 协议中 stdio 传输要求 stdout 用于 JSON-RPC，stderr 用于日志）。

确保日志级别可通过插件配置控制。

验证：启动 OpenCode，执行插件操作，检查 ~/.config/opencode/logs/ 或控制台是否包含插件的日志。
